# Conventions

How code is structured, named, and written in LastLeg. Not exhaustive — only
the things where future-me will save time by writing the rule down once.

Skim this before opening a PR. If a rule here gets in the way of something
real, change the rule and the PR.

---

## Folder structure

```
lastleg/
├── docs/                       Planning, decisions, architecture, runbooks
├── drizzle/                    Generated migration SQL + journal
├── src/
│   ├── app/                    Next.js App Router
│   │   ├── _actions/           Server actions (one file per domain)
│   │   ├── api/                Route handlers (REST-style endpoints)
│   │   ├── (auth)/             Route group for auth-gated routes (post Clerk)
│   │   ├── opengraph-image.tsx App-level conventions: OG, icon, etc.
│   │   ├── icon.tsx
│   │   ├── apple-icon.tsx
│   │   ├── layout.tsx
│   │   └── page.tsx
│   ├── components/
│   │   ├── ui/                 shadcn primitives — DON'T HAND-EDIT
│   │   └── <feature>/          Feature-grouped client + server components
│   ├── db/
│   │   ├── client.ts           Drizzle client (env-gated)
│   │   └── schema/             One file per table
│   └── lib/
│       ├── <integration>/      External-service wrappers (resend, posthog…)
│       └── <util>.ts           Pure utilities (no I/O)
└── public/                     Static assets only — no source files
```

### Rules

- **Feature-first within `components/`**, not type-first. `landing/`, `dashboard/`,
  `checkout/` — not `forms/`, `cards/`, `buttons/`.
- **One table per file** under `db/schema/`. Re-exported via the barrel
  `db/schema/index.ts`. Adding a new table: drop a file, re-export, run
  `pnpm db:generate`.
- **`components/ui/` is generated by `shadcn`** and shouldn't be hand-edited.
  If a primitive needs customising, wrap it in a non-`ui/` component.
- **No `index.ts` re-export files** except in `db/schema/`. Direct imports are
  easier to grep and to dead-code-eliminate.

## Server actions

### The "use server" rule

A file annotated `"use server"` at the top can **only export async functions**.
Next.js 16 enforces this at runtime; the build does not catch it. Exporting an
object constant — even a type-narrowed `as const` — triggers:

```
A "use server" file can only export async functions, found object.
```

…on the first invocation in production, with no useful client error.

**Pattern.** Put pure types + non-async constants in a sibling file, e.g.:

```
src/app/_actions/
├── waitlist.ts         "use server" — exports joinWaitlistAction only
└── waitlist-types.ts   exports WaitlistState type + initialWaitlistState const
```

The client component imports from both. Cost is one extra file; benefit is
no production crashes (see commit `7f719ba` for the regression).

### Shape

```ts
"use server";

export async function fooAction(
  prevState: FooState,
  formData: FormData
): Promise<FooState> {
  // 1. Parse + validate with Zod (safeParse, never throw)
  // 2. Get the (env-gated) db / external client
  // 3. Wrap the critical-path call in try/catch and return a typed error
  // 4. Best-effort fan-out (email, analytics) via Promise.allSettled
  // 5. Return { status: "ok" } | { status: "error", fieldErrors, values }
}
```

- **Never throw** — always return a discriminated-union state. Throwing
  bypasses the client's friendly error UX.
- **Validate every input with Zod** before touching the DB or any external
  service.
- **Use `Promise.allSettled` for non-critical side effects** (emails, analytics)
  so one failure doesn't break signup. Critical-path operations (DB writes,
  payment authorisations) are awaited directly inside try/catch.

## Money

**All monetary amounts are stored as integer pence in columns suffixed `_pence`.**

Never store decimals or floats for money. Conversion happens at the UI layer:

```ts
// Display
formatGBP(2400); // "£24.00"

// Parse
parseGBPInput("24"); // 2400
```

The fee formula (D007): `feePence(price) = min(100 + Math.round(price * 0.08), 400)`.
Lives in `src/lib/pricing.ts` (once created); never inline.

## Time

- **All timestamp columns are `timestamp with time zone`.** Postgres stores UTC.
- **Server-side date arithmetic uses UTC `Date` objects.** Never `new Date()`
  for anything other than "now".
- **UI displays in the user's local zone**, formatted via `Intl.DateTimeFormat`
  with `timeZone: 'Europe/London'` defaulted (since we're UK-only at MVP).
- **The user's browser timezone is the source of truth for display.**
  Don't try to infer from IP.

---

## IDs

- **All primary keys are `uuid` with `gen_random_uuid()`.** Per the Phase-1
  amendment to D-architecture: we deviated from the documented ULID choice
  because the DB-level default removes an app-side dependency. Sort-by-time
  needs go through `created_at`, not the ID.
- **Public-facing IDs in URLs**: stick with the raw UUID for now. If user
  research shows people care, we'll introduce a Sqid-style short hash later
  in a separate field — never recompute on the fly.

## Foreign keys + delete behaviour

| Relationship                                                        | onDelete   | Why                                                                                                |
| ------------------------------------------------------------------- | ---------- | -------------------------------------------------------------------------------------------------- |
| Most FKs to `users`                                                 | `restrict` | Soft-delete on users (PII redacted, row stays) per §6 GDPR. Hard-delete would orphan transactions. |
| `route_alerts.user_id → users.id`                                   | `cascade`  | Alerts are user preferences; lose them when the user goes.                                         |
| `audit_log.actor_user_id → users.id`                                | `set null` | Audit log outlives the user. Actor becomes anonymous.                                              |
| `guarantee_claims.resolved_by → users.id`                           | `set null` | Same — admin handle outlives the row.                                                              |
| FKs between domain entities (listing → user, transaction → listing) | `restrict` | We never DELETE these; status transitions handle lifecycle. Restrict prevents accidents.           |

If you find yourself wanting `cascade` between domain entities, stop and
think about whether the parent should be soft-deleted instead.

## Naming

- **Files**: `kebab-case.ts` (`waitlist-confirmation.ts`). React components are
  the exception — `PascalCase.tsx` if the file's default export is the
  component, `kebab-case.tsx` if it exports multiple things including a
  component (we use the latter throughout).
- **DB columns**: `snake_case`. Drizzle translates from the camelCase TS
  symbol.
- **DB tables**: `snake_case`, **plural** (`users`, `listings`, `transactions`).
  Exception: the waitlist table predates this rule and is `waitlist`.
- **Enum types**: `snake_case_enum` is fine. Values are also `snake_case`
  (e.g. `pending_verification`).
- **React components**: `PascalCase` symbols. File can be `kebab-case`.
- **Server actions**: end in `Action` (`joinWaitlistAction`,
  `createListingAction`). Makes them grep-able.
- **Hooks**: `use*` prefix as standard.
- **Env vars**: `SCREAMING_SNAKE_CASE`. `NEXT_PUBLIC_*` for anything that
  needs to be in the browser bundle. Document every one in `.env.example`.
- **Booleans on entities**: prefer `is_*` (`is_admin`) or `has_*`
  (`has_passenger_name`) over bare nouns.

## Path aliases

`@/` resolves to `src/`. Use it for cross-tree imports:

```ts
import { getDb } from "@/db/client";
import { Button } from "@/components/ui/button";
```

Don't use it for same-folder imports (Prettier won't shorten them and
relative imports make refactors easier).

## Env vars

- **Every env var lives in `.env.example`** with a comment explaining what
  it does and which task introduces it.
- **Server-side keys (DB URLs, API secrets) NEVER get the `NEXT_PUBLIC_`
  prefix.** That prefix bakes the value into the browser bundle — fine for
  PostHog `phc_*` keys, catastrophic for Resend `re_*` keys or DB URLs.
- **Every integration is env-gated**: if the env var is unset, the
  integration is a silent no-op. The app must boot and the landing page
  must work with zero env vars set (this is what made Phase 0 mostly
  cred-free).

```ts
// Pattern
export function getResend(): Resend | null {
  const key = process.env.RESEND_API_KEY;
  if (!key) return null;
  return new Resend(key);
}
```

Call sites handle the `null` return — TypeScript enforces this.

## Error handling

- **User-facing errors are typed states**, not thrown exceptions
  (see server-action shape above).
- **Network / SDK errors caught at the integration boundary**, logged with
  context, and converted to a typed state or an explicit no-op.
- **`console.error` for now**, Sentry later (P1-12). Always include enough
  context that grepping the log answers "what was the input?".
- **Don't swallow errors silently.** `catch (e) {}` is banned — at minimum,
  `console.error("[domain] action failed", e)`.

---

## Logging

- Prefix every log line with `[domain]` so it's filterable:
  `console.log("[waitlist] signup", { email, role })`.
- **Never log PII verbatim if avoidable.** For signup logs, the email is
  the lookup key so it's fine; for payment logs, log the Stripe ID, not
  the card.
- **Logs are append-only.** Don't try to "update" or rewrite log lines.

## Commits

- Imperative mood: "Add X", "Fix Y", "Update Z". Not "added", "fixes",
  "updating".
- First line ≤ 72 chars, body wrap at 72.
- Body explains _why_ when the diff doesn't.
- Co-author tag for Claude work (see existing commits).
- One concern per commit. If you can't summarise in one line, you have
  two commits.

## When in doubt

- **Read `DECISIONS.md`.** Most architectural ambiguity is already decided
  there.
- **Match the surrounding code.** Consistency beats personal preference.
- **Ask in a PR comment.** Don't silently introduce a new pattern.
